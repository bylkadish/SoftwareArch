# Лабораторная работа №4

**Тема:** Проектирование REST API  
**Цель работы:** Получить опыт проектирования программного интерфейса.

**Сервис:** Device Management Service (система домашней автоматизации)

*Система домашней автоматизации взаимодействует с пользователями, мобильными и веб-клиентами, а также IoT-устройствами через REST API.
Ниже представлены ключевые архитектурные решения, принятые при проектировании интерфейса.*

### **Принятые проектные решения**

1. **Использование архитектурного стиля REST**

   **Решение:**
      API реализуется в соответствии с архитектурным стилем REST (Representational State Transfer).

   **Обоснование:**

      - REST является стандартом де-факто для веб-сервисов.
      - REST легко масштабируется и хорошо подходит для облачных систем.
      - REST удобно использовать в мобильных и веб-приложениях.
      - REST хорошо интегрируется с микросервисной архитектурой.
   
   **Все операции выполняются через стандартные HTTP-методы:**

   GET - получение данных
   
   POST - создание ресурса
   
   PUT - обновление ресурса
   
   DELETE - удаление ресурса

2. **Использование JSON как формата передачи данных**

   **Решение:**
      Все данные в API передаются в формате JSON.

   **Обоснование:**

      - JSON является наиболее распространенным форматом для REST API.
      - Легко читается человеком.
      - Поддерживается всеми языками программирования.
      - Хорошо подходит для веб и мобильных клиентов.
   
   **Пример ответа для API**
```json
   {
     "id": "d8a23f",
     "name": "Living room light",
     "status": "ON",
     "firmwareVersion": "1.2"
   }
```

3. **Ресурсно-ориентированная структура URL**

   **Решение:**
      URL строятся вокруг ресурсов системы, соответствующих сущностям базы данных.

      *Основные ресурсы системы:*
      - users
      - devices
      - device-types
      - scenarios
      - events
      - notifications

   **Обоснование:**

      Это делает API:

      - Понятным.
      - Предсказуемым.
      - Соответствующим REST стандартам.

4. **Использование уникальных идентификаторов UUID**

   **Решение:**
      Все ресурсы системы используют UUID как идентификаторы.

   **Обоснование:**

      - UUID гарантирует уникальность.
      - Позволяет генерировать ID на клиенте или устройстве.
      - Удобно для распределенных систем.

5. **Использование HTTP-кодов ответа**

   **Решение:**
      API возвращает стандартные HTTP-коды.

   **Обоснование:**

      Это упрощает обработку ошибок на клиенте.

   | Код | Значение |
   |-----|----------|
   | 200 |успешный запрос|
   |201|ресурс создан|
   |400|ошибка запроса|
   |401|пользователь не авторизован|
   |404|ресурс не найден|
   |500|ошибка сервера|

6. **Использование JWT-аутентификации**

   **Решение:**
      Для доступа к API используется JWT-токен.

      *Процесс:*
      - Пользователь выполняет вход
      - Получает токен
      - отправляет его в каждом запросе

   **Обоснование:**

      - Безопасный механизм авторизации.
      - Широко используется в REST API.
      - Хорошо подходит для микросервисов.

7. **Разделение API на сервисы**

   **Решение:**
      API разделено на логические группы в соответствии с архитектурой системы.

   | Сервис | Назначение |
   |-----|----------|
   | Auth API |аутентификация пользователей|
   |Device API|управление устройствами|
   |Scenario API|управление сценариями|
   |Notification API|уведомления|
   |Video API|видеопотоки|

   **Обоснование:**

      - Соответствует микросервисной архитектуре.
      - Упрощает масштабирование.
      - Позволяет независимо развивать сервисы.

8. **Поддержка сценариев автоматизации**

   **Решение:**
      Система поддерживает создание пользовательских сценариев автоматизации.

      *Процесс:*
         Если дверь открылась, то включился свет

   **Обоснование:**
      Это ключевая функция системы домашней автоматизации.

9. **Поддержка событий устройств**

   **Решение:**
      Все устройства генерируют события (Event).

      *Примеры:*
      - Открытие дверей.
      - Движение камеры.
      - Смена температуры.

   **Обоснование:**

      - Позволяет анализировать активность системы.
      - Используется для сценариев автоматизации.
      - Позволяет пользователю просматривать историю.

10. **Система уведомлений**

    **Решение:**
      Пользователь получает уведомления при важных событиях.

      *Примеры:*
      - Открытие дверей.
      - Движение обнаружено.
      - Низкий заряд устройства.

    **Обоснование:**

      - Повышает безопасность и удобство использования системы.
