# Лабораторная работа №4

**Тема:** Проектирование REST API  
**Цель работы:** Получить опыт проектирования программного интерфейса.

**Сервис:** Device Management Service (система домашней автоматизации)

## Контекст и зона ответственности сервиса

Device Management Service отвечает за:
- регистрацию (добавление) устройства в рамках конкретного домохозяйства (household);
- хранение и изменение метаданных устройства (имя, расположение, версия прошивки, серийный номер);
- управление жизненным циклом устройства (Provisioned/Active/Inactive/Decommissioned);
- хранение статуса доступности (isOnline, lastSeenAt) — как минимум на уровне, достаточном для управления.

## **Документация по API**
### **Базовые соглашения**

**Base URL:** `/api/v1`

**Ресурсы:** домохозяйства (households) и устройства (devices).  
Устройства принадлежат домохозяйству: `householdId`

**Формат данных:** JSON (`application/json; charset=utf-8`)

**Формат ошибок:** `application/problem+json` (ProblemDetails)

**Идентификаторы:** `GUID`

### **Принятые проектные решения**

1. **Versioning: ** версионирование по пути `/api/v1/...` для неразрушающих изменений.
2. **Tenant isolation: ** все операции с устройствами строго в контексте `householdId` в URL.
3. **Разделение ответственности по компонентам: **
   - DeviceController — только HTTP/DTO/статусы.
   - DeviceRegistry / CommandProcessor / TelemetryProcessor / FirmwareUpdateManager — бизнес-логика.
4. **Команды управления как отдельный ресурс `Command`: ** отправка команды — создание ресурса команды (async модель).
5. **Статусы команд: ** `Pending -> Published` (в учебной реализации) с возможностью расширения до `Acknowledged/Failed/Timeout`.
6. **MQTT как транспорт на южную сторону: ** REST используется для внешних клиентов (через API Gateway), а к устройствам команды уходят через IoT Gateway по MQTT (в коде — интерфейс `IIotGatewayClient`).
7. **Стандартные HTTP-статусы: **
   - `201 Created` для создания устройств/команд/запросов обновления,
   - `200 OK` для чтения/обновления,
   - `204 No Content` для удаления,
   - `400 Bad Request` для ошибок параметров/валидации,
   - `404 Not Found` для отсутствующих ресурсов,
   - `409 Conflict` при попытке зарегистрировать устройство с существующим `serialNumber` внутри household.
8. **Пагинация и фильтрация: ** в списках (`page`, `pageSize`, фильтры) — масштабирование на тысячи устройств.
9. **DTO-контракты: ** вход/выход через DTO (Create/Update/Response), чтобы не раскрывать внутреннюю модель и проще версионировать.
10. **Расширяемость под “неопределённое поведение в будущем”: ** команды передают `parameters` (key-value), позволяя добавлять новые типы устройств/команд без немедленного изменения схемы БД.
